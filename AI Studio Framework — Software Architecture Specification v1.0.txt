## Глава 1. Введение

### 1.1. Назначение документа

Этот документ является основной архитектурной спецификацией проекта **AI Studio Framework**.  
Он определяет:

- цели и философию системы;
- архитектурные принципы;
- структуру кода и модулей;
- ответственность каждого компонента;
- правила расширения;
- дорожную карту развития.

Документ предназначен для всех, кто участвует в разработке, и служит единственным источником истины при принятии архитектурных решений.

---

### 1.2. Видение проекта

**AI Studio Framework** — это не Telegram-бот.

Это **платформа для мультиагентных AI-систем**, где Telegram является одним из возможных интерфейсов.

Система должна позволять:

- запускать сценарии с участием нескольких AI-агентов;
- подключать новые модели (LLM) без изменения агентов;
- расширять функциональность через инструменты и внешние API;
- работать через разные интерфейсы: Telegram, Web, REST API, CLI;
- хранить контекст диалогов, проектов и накопленных знаний.

Проект строится как расширяемое ядро, а не как монолитное приложение.

---

### 1.3. Архитектурные принципы

1. **Single Responsibility**  
   Каждый модуль решает одну задачу. Никаких «универсальных» файлов на 1500 строк.

2. **Независимость от LLM**  
   Агенты не знают, какая модель используется (GigaChat, OpenAI, Ollama и т.д.). Взаимодействие происходит через абстракцию `Provider`.

3. **Независимость от интерфейса**  
   Ядро не зависит от того, откуда пришёл запрос: из Telegram, CLI, REST API или Web.

4. **PM — не агент, а оркестратор**  
   PM (Project Manager) не пишет ответы сам. Он анализирует задачу, строит план, вызывает агентов, собирает и объединяет результаты.

5. **Расширяемость без изменения существующего кода**  
   Добавление нового агента, провайдера или инструмента не должно требовать изменения ядра системы.

6. **Event Bus как архитектурный паттерн (отложенная реализация)**  
   Компоненты общаются через события, что снижает связанность и упрощает добавление новых реакций.

---

### 1.4. Основные требования

- Работа с текстовыми запросами и генерация ответов.
- Распределение сложных задач между несколькими агентами.
- Генерация изображений.
- Хранение истории диалогов (сессионная память).
- Поддержка пользовательских команд.
- Лёгкое добавление новых агентов, моделей и интерфейсов.
- Возможность работы без постоянного интернета (опционально).

### 2.2. Потоки данных

Пример: пользователь пишет в Telegram «Создай бизнес-план для стартапа».

```
Telegram
   │
   ▼
Router: определяет, что это сложная задача
   │
   ▼
Orchestrator: принимает решение о распределении
   │
   ▼
Workflow Engine: строит план:
   Task 1: Researcher (собрать данные)
   Task 2: Strategist (разработать модель)
   Task 3: Writer (написать текст)
   │
   ▼
Агенты выполняют задачи (последовательно или параллельно)
   │
   ▼
Результаты собираются в единый ответ
   │
   ▼
Telegram: пользователь получает готовый бизнес-план
```

---

### 2.3. Модули и их ответственность

| Модуль | Ответственность |
|--------|----------------|
| **Router** | Определяет намерение пользователя (команда, ключевое слово, сложный запрос). |
| **Orchestrator** | Управляет выполнением задачи, вызывает агентов, агрегирует ответы. |
| **Workflow Engine** | Строит последовательность задач, управляет их выполнением и состояниями. |
| **Agent Registry** | Хранит всех зарегистрированных агентов, позволяет получать их по ID или роли. |
| **Agents** | Решают конкретные профессиональные задачи. Не знают о внешнем мире. |
| **Providers** | Абстрагируют доступ к LLM. Агент не знает, какая модель используется. |
| **Memory** | Хранит историю диалогов, проектов и знаний. |
| **Tools** | Утилиты (генерация изображений, SEO, работа с файлами и т.д.). |
| **Metrics** | Логирование, профилирование, мониторинг. |
| **Storage** | Постоянное хранение данных (SQLite, файлы). |

## Глава 4. Основные компоненты

### 4.1. Router

**Назначение:** понять, что хочет пользователь.

**Вход:** текст сообщения.

**Выход:** структура `Intent`, содержащая:

- тип запроса (`command`, `natural`, `complex`);
- агента (если явно указан);
- извлечённый текст;
- флаги (например, `is_urgent`).

**Правила работы:**

- Проверяет команды `/design`, `/code`, `/pm` и т.д.
- Если команда не обнаружена, анализирует ключевые слова.
- Если ключевые слова не дают результата, считает запрос сложным и передаёт Orchestrator.

**Пример:**

```python
class Router:
    def route(self, message: str) -> Intent:
        # возвращает Intent
```

---

### 4.2. Orchestrator (PM)

**Назначение:** управлять выполнением задачи.

**Вход:** Intent от Router.

**Выход:** итоговый ответ пользователю.

**Правила работы:**

- Если задача простая (прямая команда) — вызывает конкретного агента.
- Если задача сложная — запускает Workflow.
- Собирает результаты от агентов.
- Формирует единый ответ.

**Пример:**

```python
class Orchestrator:
    def execute(self, intent: Intent) -> str:
        if intent.is_simple():
            return self.call_agent(intent.agent_id, intent.query)
        else:
            return self.run_workflow(intent.query)
```

---

### 4.3. Workflow Engine

**Назначение:** строить и контролировать план задач.

**Вход:** сложный запрос пользователя.

**Выход:** массив результатов от агентов.

**Правила работы:**

- Анализирует запрос и строит цепочку задач.
- Каждая задача имеет тип, агента и параметры.
- Контролирует выполнение (успех / ошибка).
- Поддерживает последовательное и параллельное выполнение.

**Пример:**

```python
class Workflow:
    def build(self, query: str) -> list[Task]:
        # возвращает список задач
```

---

### 4.4. Agent Registry

**Назначение:** хранить всех зарегистрированных агентов.

**Методы:**

- `register(agent: BaseAgent)` — добавить агента.
- `get(agent_id: str) -> BaseAgent` — получить агента по ID.
- `all() -> list[BaseAgent]` — получить всех агентов.
- `find_by_role(role: str) -> list[BaseAgent]` — найти по роли.

**Регистрация:**

```python
registry.register(ResearcherAgent())
registry.register(StrategistAgent())
# ...
```

---

### 4.5. BaseAgent

**Назначение:** общий интерфейс для всех агентов.

**Атрибуты:**

- `id` — уникальный идентификатор.
- `name` — название.
- `role` — описание роли.

**Методы:**

- `can_handle(query: str) -> bool` — может ли агент обработать запрос.
- `execute(task: str, context: dict) -> str` — выполнить задачу.

**Агенты ничего не знают про:**

- Telegram
- HTTP
- модели LLM
- внутреннюю структуру проекта

---

### 4.6. BaseProvider

**Назначение:** абстракция для работы с LLM.

**Методы:**

- `generate(prompt: str, **kwargs) -> str` — отправить запрос и получить ответ.

**Пример:**

```python
class GigaChatProvider(BaseProvider):
    def generate(self, prompt: str) -> str:
        # вызывает GigaChat API
```

---

### 4.7. ContextManager

**Назначение:** управлять историей диалога.

**Временное поведение (Фаза 1):**

- Хранит последние N сообщений.
- По умолчанию — 20.
- Привязка к `user_id`.

---

### 4.8. Память (три уровня)

**Session Memory** — последние сообщения диалога.  
**Project Memory** — долгосрочные данные о проекте (название, цели, цвета).  
**Knowledge Memory** — результаты исследований, которые можно переиспользовать.

В **Фазе 1** реализуется только Session Memory.

---

### 4.9. Event Bus

**Назначение:** слабое связывание компонентов.

**События:**  
- `agent.started`  
- `agent.finished`  
- `workflow.completed`  
- `error.occurred`

**Реализация откладывается до Фазы 3.**

---

### 4.10. Task и Workflow

**Task** — единица работы:  
- `id`  
- `agent_id`  
- `query`  
- `status` (pending, running, done, failed)  
- `result`  
- `error`  

**Workflow** — последовательность Task с возможностью ветвления.
